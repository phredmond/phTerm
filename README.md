# phTerm Design Specification

## 1. Overview
**phTerm** is a high-performance, CRT-style terminal multiplexer built in Zig for macOS and Linux. It allows users to split their terminal into multiple panes (horizontal and vertical), spawn shells in pseudoterminals (PTYs), and handles VT100/ANSI escape sequences to emulate standard terminal behaviors.

The project avoids heavyweight GUI frameworks by running directly in the terminal, rendering its own UI using raw terminal I/O.

## 2. Architecture & Components

The application follows a poll-driven, single-threaded architecture, multiplexing input, PTY output, and system signals into a ~30 FPS render loop.

### 2.1 Main Event Loop (`main.zig`)
The `main.zig` file acts as the primary orchestrator.
- **Terminal Setup:** Enables raw mode and handles terminal restoration on exit.
- **Signal Handling:** Listens to `SIGWINCH` (window resize) and `SIGCHLD` (child process status changes) asynchronously using non-blocking flags.
- **I/O Multiplexing:** Uses POSIX `poll` to simultaneously read from `stdin` (keyboard) and the master file descriptors of all active PTY panels.
- **Event Dispatch:** Dispatches keystrokes to the layout manager or forwards them verbatim to the active child PTY.

### 2.2 Pseudoterminal Management (`pty.zig`)
Handles the low-level OS boundaries for spawning processes.
- **PTY Lifecycle:** Uses POSIX bindings (`posix_openpt`, `grantpt`, `unlockpt`, `c.ioctl`) to allocate master/slave pairs.
- **Process Forking:** Uses `fork()` and `execvp()` to spawn the shell (defaulting to `$SHELL` or `/bin/bash`), routing its `stdio` file descriptors to the PTY slave.
- **Reaping:** Exposes a `reapChildren()` block to resolve zombie states upon receiving a `SIGCHLD`.

### 2.3 Window Management & Layout Tree (`panel.zig`)
Manages multiple terminal viewports through a binary layout tree.
- **Binary Tree Layout:** Screens are split using a BSP (Binary Space Partitioning) layout. Internal nodes (`Node`) represent splits (`horizontal` or `vertical`) along with a dynamically adjustable real-number flex ratio. Leaf nodes point to a specific `Panel`.
- **Panel Struct:** Represents a single viewport containing an active PTY file descriptor, a Process ID, and an instance of the VT string emulator (`VtParser`). 
- **Tree Operations:** Support is built-in for adding, splitting (vertically/horizontally), and closing panels. Closing a panel bubbles up the tree, letting sibling nodes reclaim layout real-estate.

### 2.4 Terminal Emulator Engine (`vt.zig`)
Emulates a stateful VT-100 stream, representing the visual output of programs (vim, htop, shell) running in the PTY.
- **Parser State Machine:** Processes raw byte streams through various modes (`normal`, `escape`, `csi`, `osc`, `dcs`) to capture CSI sequences and SGR (Select Graphic Rendition) attributes.
- **Screen Buffer:** Manages a grid of `Cell` structs (`[MAX_ROWS][MAX_COLS]Cell`), where each `Cell` stores the payload character and styling information (foreground, background, bold, italics, reverse, etc.).
- **Alternate Screens:** Supports swapping between primary and alternate screens, ensuring compatibility with applications like `vim` or `less` that use an alternate buffer.
- **Scrollback:** Implements a fixed-size ring buffer for scrollback history to retain lost lines when the terminal is scrolled.

### 2.5 Input & Rendering (`input.zig`, `renderer.zig`)
- **Input Parsing:** Maps byte streams generated by terminal emulators (e.g., ANSI escape codes for arrow keys or F-keys) into actionable application commands.
- **View Rendering:** Walks the layout tree top-down, flushing cell buffers and pane borders efficiently into an off-screen string buffer, then outputting a unified diff/flush to the parent terminal.

## 3. Key Design Decisions

1. **System-level bindings instead of C dependencies:**
   Instead of using large dependencies for terminal rendering, the project actively utilizes Zig's `@cImport` capability to interface cleanly with native libc POSIX routines (`termios.h`, `sys/ioctl.h`, `fcntl.h`).
   
2. **Single-threaded Polling Loop:**
   To guarantee buffer consistency and avoid tricky race conditions on the VT emulation buffers, multiplexed I/O drives continuous updates instead of multi-threaded consumers. The 33ms target poll timeout enforces an effective 30 FPS throttle to prioritize performance without starving the CPU.

3. **In-place Screen Parsing:**
   The `VtParser` interprets escapes byte-by-byte rather than building large string arrays, mapping parsed states immediately onto an in-memory multidimensional `Cell` array.

## 4. Future Enhancements (Potential)
- Extracted key-bind mapping to support custom user configuration.
- Extended mouse/scroll integration using standard X10/SGR mouse escape sequences.
- Deeper ANSI true-color (24-bit) testing layout boundaries.
